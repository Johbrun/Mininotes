MiniNotes — Guide des vulnérabilités

11 failles classiques se cachent dans l'application. À vous de les trouver, de les exploiter, puis de les corriger.

Mode d'emploi

  • Pour chaque vulnérabilité, un défi vous est donné : c'est l'objectif concret à atteindre.
  • Si vous êtes coincé·e, vous pouvez révéler l'indice 1 (piste générale), puis l'indice 2 (piste précise).
  • Essayez de chercher sans indice d'abord, c'est là que vous apprenez le plus.
  • Une fois la faille exploitée, ouvrez le code et proposez un correctif, puis relancez l'app.

Outils utiles : votre navigateur, ses outils développeur (F12), et curl pour envoyer des requêtes personnalisées.

1

Injection SQL

Injection

Défi : connectez-vous en tant qu'administrateur sans connaître son mot de passe, en utilisant uniquement le formulaire de connexion.

1 Indice

Ce que vous tapez dans un formulaire est envoyé au serveur, qui s'en sert pour interroger la base de données. Et si le champ Nom d'utilisateur n'était pas simplement lu comme du texte, mais comme du… SQL ?

2 Indice

En SQL, le caractère ' ferme une chaîne de caractères, et -- (suivi d'un espace) commente le reste de la ligne. Essayez admin'-- comme nom d'utilisateur.

2

XSS stocké (Stored XSS)

Injection

Défi : faites apparaître une boîte d'alerte JavaScript (alert(1)) à chaque fois qu'un visiteur consulte l'une de vos notes.

1 Indice

Le contenu d'une note est affiché tel quel dans la page. Que se passe-t-il si ce contenu est du HTML… ou du JavaScript ?

2 Indice

Créez une note dont le contenu est : 

<script>alert(1)</script>

Puis ouvrez cette note. Regardez aussi ce que voit un autre utilisateur (voir la vulnérabilité n°4) qui ouvre la même note.

3

XSS réfléchi (Reflected XSS)

Injection

Défi : construisez une URL qui, lorsqu'on la visite en étant connecté, fait s'exécuter du JavaScript — sans que rien n'ait été stocké côté serveur.

1 Indice

Cherchez une page où ce que vous écrivez dans l'URL est réaffiché tel quel dans le HTML. Il y en a une facile à trouver dans la barre de navigation.

2 Indice

La page de recherche affiche votre terme : /search?q=terme. Remplacez terme par : 

<svg/onload=alert(1)>
4

IDOR — Accès aux notes d'autrui

Autorisation

Défi : lisez, modifiez ou supprimez une note qui ne vous appartient pas (par exemple celle de l'utilisateur admin), avec votre propre compte.

1 Indice

Quand vous consultez une de vos notes, regardez attentivement la barre d'adresse. Qu'y a-t-il dans l'URL ? Est-ce une donnée qu'on pourrait changer ?

2 Indice

Les URLs sont du type /note/42, /note/42/edit, /note/42/delete. Changez le nombre : le serveur vérifie-t-il vraiment que la note vous appartient ? Essayez /note/1 avec un compte qui n'est pas celui de admin.

5

Stockage faible des mots de passe

Cryptographie

Défi : en supposant que vous ayez récupéré la base de données (voir vulnérabilité n°9), retrouvez le mot de passe en clair d'un utilisateur à partir de son empreinte stockée.

1 Indice

Les mots de passe ne sont pas stockés en clair, mais sous forme d'empreinte (hash). Toutes les fonctions de hachage ne se valent pas en sécurité. Comptez les caractères d'une empreinte : cela vous dit quel algorithme a été utilisé.

2 Indice

Une empreinte de 32 caractères hexadécimaux = MD5. Cet algorithme est cassable en quelques secondes. Collez l'empreinte sur crackstation.net, ou utilisez hashcat -m 0. Regardez aussi s'il y a du sel (salt).

6

CSRF — Requête forgée

Session

Défi : fabriquez une page HTML externe à MiniNotes qui, lorsqu'une victime connectée la visite, supprime automatiquement une de ses notes.

1 Indice

Votre navigateur envoie automatiquement le cookie de session à MiniNotes pour toute requête qui part vers localhost:5000, même si la requête est déclenchée par un autre site. Est-ce que MiniNotes se protège de ça ?

2 Indice

Créez un fichier attaque.html contenant un formulaire qui poste vers la route de suppression et qui se soumet tout seul en JavaScript : 

<form id="f" method="post"
      action="http://localhost:5000/note/1/delete"></form>
<script>document.getElementById('f').submit();</script>

Ouvrez le fichier dans un navigateur où vous êtes connecté à MiniNotes. Cherchez ensuite ce qu'est un jeton CSRF.

7

Mass Assignment — Devenir admin à l'inscription

Autorisation

Défi : créez un compte qui est directement administrateur, sans jamais passer par un autre administrateur.

1 Indice

Le formulaire d'inscription ne montre que deux champs, mais la table users dans la base en contient plus (voir schema.sql). Le serveur fait-il la différence entre ce que le formulaire permet d'envoyer et ce qu'une requête HTTP peut envoyer ?

2 Indice

Envoyez vous-même une requête d'inscription avec curl, en y ajoutant un champ qui n'existe pas dans le formulaire : 

curl -X POST http://localhost:5000/register \
  -d "username=evil&password=pwn&is_admin=1"

Puis connectez-vous avec evil/pwn et allez voir /admin.

8

Redirection ouverte (Open Redirect)

Session

Défi : construisez un lien qui commence par l'URL de MiniNotes, mais qui finit par envoyer la victime sur un site que vous contrôlez (par exemple https://example.com) après sa connexion.

1 Indice

Quand vous essayez d'aller sur une page protégée sans être connecté, le site garde en mémoire où vous vouliez aller pour vous y envoyer après la connexion. Où est stockée cette destination ?

2 Indice

Regardez l'URL /login?next=.... Le serveur vérifie-t-il que la destination reste sur son propre domaine ? Essayez : 

http://localhost:5000/login?next=https://example.com
9

Traversée de chemin (Path Traversal)

Accès fichiers

Défi : téléchargez un fichier qui ne devrait pas être accessible aux utilisateurs de MiniNotes — par exemple la base de données (data/app.db) ou le code source (app.py).

1 Indice

Une route de l'application sert des fichiers depuis un dossier (uploads/). Ce dossier est-il vraiment étanche ? Un nom de fichier peut-il "remonter" dans l'arborescence ?

2 Indice

Sous Linux, .. désigne le dossier parent. Essayez d'injecter ce motif dans l'URL qui sert les fichiers uploadés : 

http://localhost:5000/uploads/../data/app.db
http://localhost:5000/uploads/../app.py

Si le navigateur refuse les ../, encodez-les : ..%2F.

10

Upload de fichier arbitraire

Accès fichiers

Défi : uploadez un fichier qui n'est pas une image, et faites-le exécuter du JavaScript quand un autre utilisateur visite son URL.

1 Indice

La page de profil permet d'uploader un avatar. L'application contrôle-t-elle vraiment que le fichier envoyé est une image ? Regardez quel type de contenu est ensuite renvoyé quand on visite /uploads/nom_du_fichier.

2 Indice

Créez un fichier xss.html contenant : 

<script>alert('XSS depuis /uploads/')</script>

Uploadez-le comme avatar, puis visitez http://localhost:5000/uploads/xss.html. Le navigateur l'interprète comme une vraie page HTML.

11

Contrôle d'accès défaillant (/admin)

Autorisation

Défi : accédez à la page d'administration (/admin) avec un compte qui n'est pas administrateur.

1 Indice

La page /admin existe (elle est mentionnée dans le README). Qu'est-ce qui est censé vous empêcher d'y aller avec un compte lambda ? Le site vérifie-t-il seulement que vous êtes connecté, ou bien aussi quel rôle vous avez ?

2 Indice

Créez un compte ordinaire, connectez-vous, puis tapez directement http://localhost:5000/admin dans la barre d'adresse. Que voyez-vous ? Cherchez dans app.py la différence entre "être connecté" et "être admin".

Une fois toutes les failles trouvées et exploitées, proposez un correctif pour chacune et vérifiez que le défi ne fonctionne plus.
